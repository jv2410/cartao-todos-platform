# Story 1.5: Session Management and Auto-Logout

## Status: Completed

## Description
As a system, I need to manage user sessions securely with automatic expiration, so that inactive sessions don't pose a security risk.

## Acceptance Criteria
- [x] Given I log in successfully, when my JWT token is created, then it expires after 8 hours (28800 seconds) by default
- [x] Given I check "Remember me" during login, when my session is created, then the JWT token expires after 30 days (2592000 seconds)
- [x] Given I have an active session, when I perform any user action (click, API call, navigation), then my session expiration timer is extended by 8 hours (automatic activity-based extension)
- [x] Given my session has 5 minutes remaining, when the timer reaches 5:00, then I see a warning dialog: "Your session will expire in 5 minutes. Do you want to extend your session?"
- [x] Given I see the session warning dialog, when I click "Extend Session", then my session is extended by 8 hours and the warning disappears
- [x] Given I see the session warning dialog, when I do nothing for 5 minutes, then I am automatically logged out and redirected to /login with message "Your session has expired. Please log in again."
- [x] Given I click "Logout" in the navigation, when I confirm logout, then my session is destroyed (JWT invalidated in Redis) and I am redirected to /login
- [x] Given I am logged out, when I try to access a protected route (e.g., /dashboard, /settings), then I am redirected to /login with message "Please log in to access this page"
- [x] Given I have multiple browser tabs open, when my session expires in one tab, then all tabs detect the expiration and redirect to /login
- [x] Given I close my browser, when I return within 8 hours (or 30 days with "remember me"), then my session is still active and I can continue working

## Technical Notes

### Session Storage
- **Redis Cache**:
  - Key: `session:{token_hash}` (hash of JWT token for privacy)
  - Value: JSON object `{ user_id, username, created_at, expires_at, device_type }`
  - TTL: 8 hours (28800 seconds) or 30 days (2592000 seconds) with "remember me"

- **JWT Token**:
  - Payload: `{ user_id, username, iat (issued at), exp (expiration) }`
  - Algorithm: HS256
  - Secret: Loaded from environment variable `JWT_SECRET`
  - Stored as httpOnly cookie in browser

### API Endpoints
- **POST /api/auth/logout**
  - Headers: `Authorization: Bearer {token}`
  - Response: `{ success: boolean }`
  - Logic: Delete session from Redis, clear httpOnly cookie
  - Status codes: 200 (success), 401 (unauthorized)

- **POST /api/auth/extend-session**
  - Headers: `Authorization: Bearer {token}`
  - Response: `{ success: boolean, new_expires_at: ISO8601 }`
  - Logic: Extend session TTL in Redis by 8 hours, return new expiration time
  - Status codes: 200 (success), 401 (unauthorized or expired)

- **GET /api/auth/session-status**
  - Headers: `Authorization: Bearer {token}`
  - Response: `{ active: boolean, expires_at: ISO8601, expires_in_seconds: number }`
  - Logic: Check session exists in Redis, return remaining time
  - Status codes: 200 (success), 401 (session expired or invalid)

### Frontend Session Management
- **Session Monitor** (`/frontend/src/lib/sessionMonitor.ts`):
  - Poll `/api/auth/session-status` every 60 seconds
  - Calculate time remaining until expiration
  - Trigger warning dialog when 5 minutes remaining
  - Trigger logout when session expires

- **Activity Tracker** (`/frontend/src/lib/activityTracker.ts`):
  - Listen for user activity events: click, keypress, scroll, mousemove
  - Debounce activity events (only extend session once per minute max)
  - Call `/api/auth/extend-session` when activity detected

- **Session Warning Dialog** (`/frontend/src/components/session-warning-dialog.tsx`):
  - Modal dialog with countdown timer (5:00, 4:59, 4:58, ...)
  - "Extend Session" button (calls `/api/auth/extend-session`)
  - "Logout" button (calls `/api/auth/logout`)
  - Cannot be dismissed without action (modal backdrop click disabled)

- **Multi-Tab Synchronization**:
  - Use localStorage to broadcast session expiration across tabs
  - When one tab logs out or expires, all tabs detect and redirect to /login
  - Implementation: localStorage event listener on "session_status" key

### Activity-Based Session Extension
- **Trigger Events**: click, keypress, API call (any authenticated request)
- **Extension Logic**:
  1. Detect user activity via event listeners
  2. Check last extension timestamp (stored in memory)
  3. If more than 1 minute since last extension, call `/api/auth/extend-session`
  4. Update session TTL in Redis to 8 hours from now
  5. Store new expiration time in memory for countdown calculation

- **Rate Limiting**: Extend session maximum once per minute (prevent excessive Redis updates)

### Session Invalidation on Logout
- **Backend**: Delete session key from Redis (`DEL session:{token_hash}`)
- **Frontend**: Clear httpOnly cookie, clear localStorage session data, redirect to /login
- **Audit Log**: Create entry with action: "USER_LOGOUT", user_id, timestamp, IP address

### Protected Route Middleware (Next.js)
- **Middleware** (`/frontend/src/middleware.ts`):
  - Check if route requires authentication (all routes except /login)
  - Verify JWT token exists in request cookies
  - Verify session exists in Redis (call backend or check locally)
  - If no valid session, redirect to /login with message
  - If force_password_change is true, redirect to /change-password (from Story 1.2)

## Notes
- Session expiration warning at 5 minutes provides sufficient time for user to decide (not too early, not too late).
- Activity-based extension ensures users don't get logged out while actively working (improved UX).
- Multi-tab synchronization is critical for users with multiple tabs open (common in admin workflows).
- httpOnly cookies prevent JavaScript access to JWT token (XSS protection).
- Redis TTL automatically expires sessions (no manual cleanup required).
- "Remember me" extends session to 30 days (useful for trusted devices, e.g., admin's personal computer).
- Session monitoring polls every 60 seconds (balance between responsiveness and API load).
- Activity debouncing (once per minute) prevents excessive session extension requests.
- Audit logging for logout events provides transparency and compliance (LGPD).
- Consider adding "Last Activity" timestamp in user profile (Phase 2 feature for monitoring inactive accounts).
