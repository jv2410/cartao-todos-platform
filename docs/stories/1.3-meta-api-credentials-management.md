# Story 1.3: Meta API Credentials Management

## Status: Completed

## Description
As an admin, I want to securely configure Meta API credentials in a protected settings section, so that the platform can connect to WhatsApp Business API on my behalf.

## Acceptance Criteria
- [x] Given I am authenticated, when I navigate to /settings, then I see the Settings page with a "Meta API Configuration" section
- [x] Given I am on the Settings page, when I view the Meta API Configuration section, then I see form fields for: WABA ID, Phone Number ID, Access Token, and Business Manager ID (optional)
- [x] Given I have not configured credentials yet, when I view the form, then all fields are empty with placeholder text explaining what each field is
- [x] Given I am entering an Access Token, when I type in the field, then the text is masked by default (showing dots or asterisks)
- [x] Given the Access Token field is masked, when I click the "Show/Hide" toggle (eye icon), then the token is revealed in plain text (and masked again when clicked again)
- [x] Given I have filled in all required fields (WABA ID, Phone Number ID, Access Token), when I click "Save Credentials", then a confirmation dialog appears: "Are you sure you want to save these credentials? Existing credentials will be overwritten."
- [x] Given I confirm the save action, when credentials are saved, then they are encrypted using AES-256 before being stored in the database
- [x] Given credentials are successfully saved, when the save operation completes, then I see a success toast notification: "Credentials saved successfully"
- [x] Given credentials are saved, when an audit log entry is created, then it includes: user_id, action: "CREDENTIALS_UPDATED", timestamp, IP address, and changes (masked token showing only last 4 characters)
- [x] Given I have previously saved credentials, when I return to the Settings page, then I see the WABA ID and Phone Number ID populated, but the Access Token is masked showing only the last 4 characters (e.g., "••••••••1234")
- [x] Given I want to view my saved Access Token, when I click "Show Token" button, then the full token is decrypted and displayed (with "Hide Token" button to mask again)
- [x] Given I update any credential field, when I click "Save Credentials", then a warning appears if credentials have already been tested successfully: "You are updating tested credentials. You may need to test the connection again."

## Technical Notes

### Database Tables
- **api_credentials** table (from architecture):
  - `id` (UUID, primary key)
  - `user_id` (UUID, foreign key to users)
  - `waba_id` (string) - WhatsApp Business Account ID
  - `phone_number_id` (string) - Phone Number ID
  - `encrypted_access_token` (text) - AES-256 encrypted access token
  - `business_manager_id` (string, nullable) - Optional Business Manager ID
  - `last_tested_at` (timestamp, nullable) - Set when connection test succeeds (Story 1.4)
  - `created_at` (timestamp)
  - `updated_at` (timestamp)

### API Endpoints
- **GET /api/settings/credentials**
  - Headers: `Authorization: Bearer {token}`
  - Response: `{ waba_id: string, phone_number_id: string, access_token_masked: string (last 4 chars), business_manager_id?: string, last_tested_at?: ISO8601 }`
  - Status codes: 200 (success), 401 (unauthorized), 404 (no credentials found)

- **PUT /api/settings/credentials**
  - Headers: `Authorization: Bearer {token}`
  - Request body: `{ waba_id: string, phone_number_id: string, access_token: string, business_manager_id?: string }`
  - Response: `{ success: boolean, message: string }`
  - Status codes: 200 (success), 400 (validation error), 401 (unauthorized), 500 (encryption error)

- **POST /api/settings/credentials/decrypt-token**
  - Headers: `Authorization: Bearer {token}`
  - Response: `{ access_token: string }` (full decrypted token)
  - Status codes: 200 (success), 401 (unauthorized), 404 (no credentials found)

### Encryption Implementation
- **Encryption Algorithm**: AES-256-GCM (Galois/Counter Mode for authenticated encryption)
- **Encryption Key**: Loaded from environment variable `ENCRYPTION_KEY` (32-byte hex string, never committed to git)
- **Encryption Process**:
  1. Generate random initialization vector (IV) for each encryption
  2. Encrypt access token using AES-256-GCM with key and IV
  3. Store encrypted token + IV (concatenated or separate fields)
  4. Return encrypted text as base64 or hex string

- **Decryption Process**:
  1. Retrieve encrypted token and IV from database
  2. Decrypt using AES-256-GCM with encryption key
  3. Return plain text access token
  4. Only decrypt when explicitly requested (show token, test connection)

### Security Requirements
- **Encryption Key Management**:
  - Encryption key stored in environment variable (never in code or database)
  - Key must be 32 bytes (256 bits) for AES-256
  - Generate key once during initial setup: `openssl rand -hex 32`
  - Key rotation not required for MVP (consider for Phase 2)

- **Access Control**:
  - Only authenticated admins can view/update credentials
  - Audit log captures every credential access (view, update)
  - Access Token never returned in API responses (only masked version)
  - Full token only decrypted when explicitly requested via separate API call

- **Validation**:
  - WABA ID: Numeric string, 15-17 digits
  - Phone Number ID: Numeric string, 15-17 digits
  - Access Token: Alphanumeric string, minimum 50 characters (Meta tokens are typically 180-200 chars)
  - Business Manager ID: Numeric string, 15-17 digits (optional)

### Audit Logging
- **Event**: CREDENTIALS_UPDATED
- **Captured Data**: user_id, action, timestamp, IP address, changes (masked token: "••••1234")
- **Event**: CREDENTIALS_VIEWED
- **Captured Data**: user_id, action, timestamp, IP address (when admin clicks "Show Token")
