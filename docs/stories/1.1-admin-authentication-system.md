# Story 1.1: Admin Authentication System

## Status: Implemented

## Description
As an admin, I want to log in with a secure password, so that only authorized personnel can access the messaging platform and protect our Meta API credentials.

## Acceptance Criteria
- [x] Given I am on the login page, when I enter valid username and password, then I am authenticated and redirected to the dashboard
- [x] Given I am a new admin, when I use the default password "100101", then I am authenticated but immediately redirected to password change screen
- [x] Given I enter incorrect credentials, when I submit the login form, then I see a clear error message "Invalid username or password"
- [x] Given I have failed login 5 times, when I attempt a 6th login, then my IP is locked out for 15 minutes with message "Too many failed attempts. Try again in X minutes."
- [x] Given I successfully log in, when I close my browser and return within 8 hours, then my session is still active
- [x] Given I check "Remember me", when I return after 8 hours, then my session extends to 30 days
- [x] Given I click the password visibility toggle, when the eye icon is clicked, then the password field switches between masked and visible text
- [x] Given I submit the login form, when authentication is in progress, then I see a loading spinner and the button text changes to "Entrando..."
- [x] Given my session is active, when the JWT token is set, then it is stored as httpOnly cookie to prevent XSS attacks
- [x] Given I have an active session, when I navigate directly to /login, then I am redirected to /dashboard

## Technical Notes

### Database Tables
- **users** table (from architecture):
  - `id` (UUID, primary key)
  - `username` (string, unique)
  - `password_hash` (string, bcrypt hashed)
  - `force_password_change` (boolean, default true for new users)
  - `created_at` (timestamp)
  - `updated_at` (timestamp)

- **sessions** table (from architecture):
  - `id` (UUID, primary key)
  - `user_id` (UUID, foreign key to users)
  - `token` (string, JWT token hash)
  - `expires_at` (timestamp)
  - `device_type` (string)
  - `created_at` (timestamp)

### API Endpoints
- **POST /api/auth/login**
  - Request body: `{ username: string, password: string, remember_me?: boolean }`
  - Response: `{ token: JWT, expires_in: 28800, user: { id, username, force_password_change } }`
  - Status codes: 200 (success), 401 (invalid credentials), 429 (rate limited), 423 (account locked)

- **GET /api/auth/me**
  - Headers: `Authorization: Bearer {token}`
  - Response: `{ id, username, force_password_change: boolean }`
  - Status codes: 200 (success), 401 (unauthorized)

### Frontend Components
- **Login Page** (`/frontend/src/app/login/page.tsx`):
  - Form with username and password fields
  - Remember me checkbox
  - Password visibility toggle
  - Loading state management
  - Error message display

- **Components** (from front-end spec):
  - FormField component (Label + Input + Error message)
  - Button component (Primary variant with loading state)
  - Input component (Password type with visibility toggle)

### Security Requirements
- **Password Hashing**: bcrypt with cost factor 10
- **JWT Token**: HS256 algorithm, 8-hour expiration (28800 seconds), 30-day for "remember me"
- **Cookie Security**: httpOnly, Secure (HTTPS only), SameSite=Strict
- **Rate Limiting**: 5 failed attempts per IP within 15 minutes (tracked in Redis)
- **Session Storage**: JWT tokens stored in Redis cache for validation and revocation

### Redis Cache Keys
- `login_attempts:{ip_address}` - Counter with 15-minute TTL
- `account_lockout:{ip_address}` - Boolean flag with 15-minute TTL
- `session:{token_hash}` - Session data with 8-hour or 30-day TTL

## Testing Requirements

### Unit Tests
- **authService.ts**:
  - Test login with valid credentials returns JWT token
  - Test login with invalid credentials returns null
  - Test bcrypt password comparison works correctly
  - Test default password "100101" authenticates successfully
  - Test force_password_change flag is checked correctly

- **rateLimitService.ts**:
  - Test failed login increments Redis counter
  - Test 5th failed login triggers account lockout
  - Test lockout expires after 15 minutes (TTL)
  - Test successful login resets failed attempt counter

- **jwt.ts**:
  - Test JWT token generation includes user_id and username
  - Test JWT token verification validates signature
  - Test expired JWT token returns error
  - Test remember_me flag extends expiration to 30 days

### Integration Tests
- **POST /api/auth/login**:
  - Test successful login with valid credentials returns 200 and JWT token
  - Test login with invalid username returns 401
  - Test login with invalid password returns 401
  - Test 6th failed login returns 423 (locked) status
  - Test remember_me flag sets 30-day expiration in response
  - Test httpOnly cookie is set in response headers

- **GET /api/auth/me**:
  - Test with valid JWT token returns user data (200)
  - Test with invalid JWT token returns 401
  - Test with expired JWT token returns 401
  - Test without Authorization header returns 401

### Edge Cases
- Test login with empty username or password returns validation error
- Test login with SQL injection attempt is sanitized (parameterized queries)
- Test login with extremely long username/password (>1000 chars) is rejected
- Test concurrent login attempts from same IP increment counter correctly (Redis atomic operations)
- Test session persists across browser tab close (if remember_me is true)
- Test password visibility toggle does not affect form submission
- Test login form is accessible via keyboard navigation (Tab, Enter)

### Security Tests
- Verify password is hashed with bcrypt (never stored as plain text)
- Verify JWT secret is loaded from environment variable (not hardcoded)
- Verify httpOnly cookie prevents JavaScript access to token
- Verify Secure flag ensures cookie only sent over HTTPS
- Verify SameSite=Strict prevents CSRF attacks
- Verify rate limiting prevents brute force attacks
- Verify account lockout resets after 15 minutes

## Notes
- This is the foundational story for the entire platform. All other features depend on secure authentication.
- Default password "100101" is intentionally simple for MVP. Consider more secure initialization in production.
- Rate limiting is IP-based for MVP. Consider user-based rate limiting in Phase 2 for multi-user environments.
- Session management uses Redis for fast lookup and automatic expiration via TTL.
- JWT tokens are stored as httpOnly cookies (not localStorage) to prevent XSS attacks.
